## 1. 愿景与问题定义

### 1.1 问题

数以百万计的青少年用过 ChatGPT，他们知道 AI 能写代码。但他们从未把代码变成一个活的、可分享的产品。从「AI 生成了代码」到「我有一个能用的东西」之间的鸿沟，对从未打开过终端、运行过命令或部署过任何东西的人来说，是隐形但不可逾越的。

现有的 Vibe Coding 工具（Cursor、Bolt、Replit）为成年人解决了这个问题，但它们默认用户已有软件开发的心智模型。没有任何产品通过一种有趣的、渐进的体验，专门为青少年教授 Vibe Coding 工作流。

### 1.2 愿景

CodeTown 是一个基于浏览器的游戏，玩家通过创建真实的软件项目来建设一座虚拟小镇。小镇上的每栋建筑都是玩家通过引导式 Vibe Coding 创建的真实可用的 Web 项目（网站、工具、游戏、应用）。在 4 个关卡中，玩家逐步学习 AI 辅助软件创作的 9 个核心概念——从写出第一条需求到编排多 Agent 协作构建并对接外部 API。

> **核心设计原则：** 每个概念都是在玩家遇到真实需要时才引入的——永远不会提前用教程解释。学习永远是创造的副产品。

### 1.3 目标用户

**主要用户：** 12–18 岁初高中生。用过 ChatGPT 聊天，知道 AI 能写代码，但从未自己运行过代码。驱动力是「我想做个酷东西给同学看」，而不是「我想学编程」。

**次要用户：** 对 Vibe Coding 好奇但被开发者工具吓退的成年人。

### 1.4 产品形态

Web App（浏览器直接访问，无需安装）。中文为主，面向国内初高中生。仅桌面端（1280px+）。进入小镇需登录（支持邮箱或手机号+密码）。

### 1.5 参考产品与竞争定位

**Elisa**（github.com/zoidbergclawd/elisa）——一个面向 8–14 岁儿童的 Electron 桌面应用，通过 Blockly 积木编排 AI Agent 团队。CodeTown 从 Elisa 汲取了核心 AI 工程概念体系，但在以下维度进行了根本性差异化：

| 维度 | Elisa | CodeTown |
|------|-------|----------|
| 运行环境 | Electron 桌面应用，需安装 Node.js 20+ | 纯 Web App，浏览器直接打开 |
| 编辑范式 | Blockly 积木式编程（开发者心智模型） | 设计图纸 + 文件夹（创意工作者心智模型） |
| 导航架构 | 标签式 IDE（Workspace / Mission Control） | 小镇地图 → 设计桌 → 工地 |
| 学习路径 | 无引导，14 个概念全部同时可用 | 4 关卡渐进解锁 9 个组件 |
| 构建可见性 | 抽象 DAG + 空洞叙事 | 工匠角色工位 + 翻转卡片 Teaching Moments |
| 社交层 | 无 | 小镇主页、项目分享链接 |
| 成本显示 | 美元 "$0.3924" | 人民币 "¥0.28" |
| API 密钥 | 环境变量，用户自备 | 引导式 BYOK，"建筑许可证"比喻 |
| 数据持久化 | 内存 + JSON crash recovery | Supabase（Auth + Postgres + Storage） |
| 代码执行 | 本地进程（Node/Python） | 沙盒 iframe（sandbox="allow-scripts"） |

**CodeTown 核心壁垒：** 零安装即体验、工匠角色可见性、渐进式教育设计、社交即产品、设计图纸隐喻。

### 1.6 非目标（Non-Goals）

以下功能明确不在 v6.5 范围内，AI 实现时不应自行添加：

**项目管理**
- ❌ 删除已完成的项目（只能通过重新构建覆盖）
- ❌ 项目版本历史 / 回滚到旧版本
- ❌ 同一关卡保存多个项目变体

**代码交互**
- ❌ 查看生成的 HTML/CSS/JS 源码
- ❌ 导出 / 下载代码文件
- ❌ 在 CodeTown 内手动编辑代码
- ❌ 代码语法高亮 / 代码编辑器 UI

**图书馆输入**
- ❌ 粘贴 URL 自动抓取网页内容
- ❌ 从云盘（Google Drive / 百度网盘等）导入文件
- ❌ PDF 文件解析
- ❌ 上传内容审核 / 敏感词过滤

**社交功能**
- ❌ 访客评论 / 点赞 / 收藏
- ❌ 用户互访小镇（只能通过分享链接访问单个项目）
- ❌ 排行榜 / 社区发现页 / 推荐流
- ❌ 关注 / 粉丝系统

**账号与认证**
- ❌ 游客模式（进入小镇即需登录）
- ❌ 账号注销 / 数据删除功能
- ❌ 第三方登录（微信/Google 等）

**工匠定制**
- ❌ 用户直接编辑工匠的 System Prompt（仅可填写 project-based prompt）
- ❌ 用户查看工匠的完整 Prompt（只看到名称 + 职责描述 + 自己填的 project-based prompt）
- ❌ 导入 / 导出工匠配置
- ❌ 自定义新角色（仅可使用四个基础角色）

**其他**
- ❌ 国际化 / 英文版
- ❌ 自由建造模式的详细设计（v6.5 仅实现入口）
- ❌ 离线模式 / PWA
- ❌ 移动端适配（不支持移动端，仅桌面端 1280px+）
- ❌ 音效自定义（仅开关）
- ❌ 成就徽章系统（移至 V2）
- ❌ 回放模式 / Demo 安全网

## 2. 九大组件体系

| # | 小镇名称 | AI 工程原语 | 功能 | 首次出现 |
|---|---------|------------|------|---------|
| 1 | 📋 委托单 | System Prompt + Template | 要做什么 + 项目类型 | 第 1 关 |
| 2 | 🎨 装修方案 | Style Tokens + Persona | 视觉预设 + 语气 | 第 1 关 |
| 3 | 📝 需求清单 | Prompt Constraints + Acceptance Criteria | 功能/约束/When-Then/数据 | 第 1 关 |
| 4 | 🤖 工匠团队 | Multi-Agent Roles | 规划者/建造者/测试者/审查者（四个固定角色） | 第 1 关(自动)，第 2 关(可配置)，第 3 关(完全自由) |
| 5 | 🎒 技能包 | Reusable Prompts / Skills | 给工匠装备可复用能力 | 第 3 关 |
| 6 | 📜 规则系统 | Trigger-based Hooks / Rules | 条件触发的自动约束 | 第 3 关 |
| 7 | 🔀 流程控制 | Flow Control | 先后/并行/迭代/人工审批 | 第 3 关 |
| 8 | 🔌 Plug | External API / MCP | 连接外部 API | 第 3 关(预制)，第 4 关(+魔法连接) |
| 9 | 📚 图书馆 | RAG / Reference Context | 粘贴文本/上传文件注入 Agent | 第 4 关 |

> **部署不是独立组件。** 每个项目自动发布到小镇地图并生成分享链接。
> **测试不是独立组件。** 测试者（Tester）是工匠团队中的一个角色，验收行为内置于工匠执行流程。
> **施工计划（Task DAG）不是独立组件。** Planner Agent 拆解任务生成 DAG，玩家通过流程控制组件下达执行约束。

### 2.1 委托单（Commission）

设计图纸第一个卡槽。自由文本（做什么）+ 项目类型下拉（网站/工具/游戏/应用）。前期预填，后期自写。技术上成为所有 Agent 的顶层 system prompt。

### 2.2 装修方案（Style）

两张子卡片：外观（4-5 张预设缩略图选择）+ 性格（自由文本影响语气）。外观→CSS 方向指导，性格→tone 指令。

### 2.3 需求清单（Requirements）

四种子类型逐步解锁：功能（L1）、约束（L2）、当/则（L2）、数据需求（L4）。每种有不同边框颜色和 placeholder 句式。

> **说明：** L3 不新增子类型（L3 重点是技能包/规则/流程控制/Plug）。

### 2.4 工匠团队（Craftsmen）

**四个固定基础角色：**

| 角色 | 功能 | 首次出现 |
|------|------|---------|
| 📐 规划者（小策） | 把项目需求拆解为具体实现步骤（生成 Task DAG） | L1 自动 |
| 🏗️ 建造者（小匠） | 写代码 | L1 自动 |
| 🧪 测试者（小检） | 验证需求（行为测试） | L2 默认 |
| 📋 审查者（小审） | 审查质量 | L3 可选 |

每个工匠 = 一次独立 Claude API 调用，不同 system prompt。

**Project-based Prompt：** 用户点击角色卡片后，可以在卡片上的文本框中填写针对该角色的项目级描述。这段描述会自动注入该工匠的 system prompt，让工匠在本项目中具有特定的行为或关注点。

**解锁节奏：**

- **L1：** Planner + Builder 自动出现，用户不能修改团队配置。用户第一次看到"先规划再建造"的基本流程。
- **L2：** Planner + Builder + Tester 默认配置，用户可以调整（加减四个基础角色中的任意角色）。这是用户第一次接触团队配置界面，但有一个安全的推荐起点。
- **L3-L4：** 全部四个基础角色可用，完全自由增删配置。用户可以为每个角色填写 project-based prompt。

**Planner 与 Meta Planner 的区别：**

- **Meta Planner**（系统级，用户可见结果）：用户点击"制定工作计划"后运行。负责整理图纸上的卡片——按卡槽从上到下的顺序排列，组织项目输入。完成后按钮变为"开始建造"。
- **Planner**（工匠角色，用户可见）：在工地上第一个亮起的工位。拿到整理好的项目需求后，拆解为具体实现步骤（"1. 搭 HTML 骨架 → 2. 写样式 → 3. 加产品列表 → 4. 做响应式"），输出 Task DAG，其他工匠再按 DAG 执行。

测试者负责"当 X → 则 Y"验收条件验证，通过/失败展示在工地工位上。

### 2.5 技能包（Skills）

给工匠装备的可复用能力。每张技能卡片两个字段：目标工匠（建造者/测试者/审查者/全部）+ 技能内容。装备后注入对应工匠的 system prompt。

与规则系统的区别：技能是**主动装备的能力**（"这个工匠会做什么"），规则是**被动触发的约束**（"什么时候必须遵守什么"）。小镇比喻：技能 = 工匠的技能证书，规则 = 贴在墙上的镇规告示。

内置模板：📱 响应式布局、🌙 暗黑模式、♿ 无障碍设计、💬 代码注释、🛡️ 代码整洁。

### 2.6 规则系统（Rules）

条件触发的自动约束。一张卡片两个字段：触发时机（任务完成时/测试失败时/部署前）+ 规则内容。触发后自动注入对应阶段的 Agent prompt。

**触发粒度：**
- **任务完成时：** 每个子任务完成都会触发，而非所有任务完成后触发一次。这样用户可以配置"每完成一个任务就让审查者看一下"这种规则。

内置模板：🔄 测试失败自动重试（max 3）、📋 任务完成后代码审查、🛡️ 部署前安全检查。

### 2.7 流程控制（Flow Control）

玩家对 Planner Agent 生成的 Task DAG 下达的执行约束。流程控制是**独立的指令卡片**，用户从文件夹拖入设计图纸。

**卡片结构：**

```
┌─────────────────────────────────────┐
│  🔀 流程控制                         │
├─────────────────────────────────────┤
│  指令类型：[ 同时进行 ▼ ]            │
│                                     │
│  任务 1：[ 展示新品    ▼ ] [ - ]    │
│  任务 2：[ 有价格标签  ▼ ] [ - ]    │
│                        [ + 添加任务 ]│
└─────────────────────────────────────┘
```

- **指令类型下拉：** 先后 / 同时进行 / 持续改进 / 让我看看
- **任务选择器：** 下拉选项来自需求清单中的条目
- **增减任务：** 用户可以通过 +/- 按钮增减任务数量

**四种指令：**

| 指令 | 含义 | 工地表现 |
|------|------|---------|
| 🔗 先/后（First → Then） | 顺序依赖 | 工位线性排列 |
| ⚡ 同时进行（Do at the same time） | 并行执行 | 传送带分叉再合并 |
| 🔄 持续改进（Keep improving） | 迭代直到满意（max 3） | 工位循环箭头 |
| 👀 让我看看（Check with me） | 人工审批节点 | 工位暂停等待玩家 |

**技术实现：** 流程控制指令作为约束条件注入 Planner Agent，Planner 在生成 Task DAG 时会考虑这些约束。如果约束不合理（如有依赖冲突），Planner 可以忽略或调整。

### 2.8 Plug

连接外部世界的接口。小镇比喻：给项目接上外部能力的接口铺。

**L3（预制 Plug）：** 2 个预配置 MCP，用户从文件夹中选择拖入设计图纸：

- 🖼️ **图片素材**（Unsplash）
  - 卡片上只有一个文本框，placeholder："输入关键词，如'面包'、'星空'"
  - 构建时 Agent 拿到关键词，调用 Unsplash API 搜索，返回图片 URL 嵌入项目
  
- 🗺️ **地图**（Mapbox）
  - 卡片上只有一个文本框，placeholder："输入地点，如'北京天安门'、'东京塔'"
  - 构建时 Agent 拿到地点名，调用 Mapbox Geocoding API 获取坐标，生成带标记的静态地图图片 URL

后端用平台 Key 代理，用户无需自行注册。

**L4（+魔法连接）：** 保留 L3 的 2 个预制 Plug，额外新增"魔法连接"：

- 🪄 **魔法连接**
  - 卡片上只有一个文本框，placeholder："描述你想要什么外部信息"
  - 用户可以自由输入，如："上海今天的天气"、"把这段话翻译成英文"、"搜索关于猫的新闻"
  - 后端有分类器（用 Claude），把用户描述匹配到预设的 API 类别（天气/翻译/搜索/地图等）
  - 实际调用白名单内的 API，安全可控
  - 如果匹配不到，返回"暂时不支持这种连接"

**设计理由：** 魔法连接让用户用自然语言描述需求，每个用户都能玩出不同花样，同时后端仍然走白名单 API，确保安全。比单纯增加几个预制 Plug 更有趣，比开放自定义 URL 更安全。

### 2.9 图书馆（Library）

让工匠先查资料再干活。两种输入方式并列：

- **粘贴文本框（主路径）：** 用户直接粘贴参考资料文本，每条粘贴成为一条"资料"。**单条粘贴限制：≤2000 字符。**
- **上传文件（补充路径）：** 支持 .txt / .md / .doc / .docx，不支持 PDF。上传后自动提取文本。单文件 ≤500KB。

所有资料扁平展示（无文件夹、无分类）。后端提取文本 → 注入 Agent prompt `<reference_documents>`。总注入 ≤8000 tokens。

小镇比喻：工匠去图书馆借了资料，开工前先读一遍。

## 3. 设计桌（Workspace）

### 3.1 设计理念：设计图纸 + 文件夹

桌面上铺着设计图纸，图纸有标注好的卡槽。左侧是一排彩色文件夹，每个文件夹对应一个组件。

- **结构即教学：** 卡槽标签教会玩家「一个项目需要什么」
- **渐进披露：** 每关图纸模板不同（L1: 3 槽，L4: 9 槽）
- **磁吸入槽：** 拖卡片到图纸，匹配槽高亮并吸入；拖错无反应
- **图纸滚动：** 木纹桌面固定，图纸纵向滚动
- **非线性填写：** 用户在填写卡片时只需要考虑功能，不用考虑顺序

### 3.2 Tab 导航与两栏布局

**右上角 Tab 切换：** 两个长方形按钮标签——「🗒️ 设计桌」和「🏗️ 工地」。进入时默认在设计桌 Tab。工地 Tab 在点击"开始建造"之前为灰色不可切换。建造开始后两个 Tab 可自由切换，但设计桌内容变为只读（不可编辑卡片）。

**左栏（200px）文件夹：** 9 个彩色文件夹标签，纵向排列。每个文件夹对应一个组件（委托单、装修方案、需求清单……）。未解锁文件夹灰色+🔒。点击文件夹展开露出里面的卡片选项（预设模板/空白卡片）。

**中栏（flex-1）设计图纸：** 木纹桌面固定，蓝色网格图纸纵向滚动。顶部 sticky 项目概要（名称+类型+已填写计数）。底部按钮。

**无右栏预览。** 预览在工地揭幕时刻出现。设计桌 = 规划，工地 = 构建+预览。

**卡槽类型：** 必填（实线+脉冲）、可选（虚线）、锁定（灰色+🔒+"第X关解锁"）。

**底部按钮两阶段：**

1. **"📐 制定工作计划"**（必填卡槽全部填完后可点击）→ Meta Planner 运行 → 图纸上的卡片按卡槽从上到下的顺序排列整齐。
2. **"🚀 开始建造"**（整理完成后按钮自动切换）→ 点击后进入工地 Tab，Planner 工匠开始拆解 Task DAG。

## 4. 工地（Construction Site）

### 4.1 核心理念

工匠工位替代 Elisa 的抽象 DAG。气泡替代空洞 Narrator。翻转卡片替代 Learn 标签页。

### 4.2 工地布局（左右分栏）

**左侧 65%：Agent 团队区** — 工位卡片竖向排列，从上到下依次是 Planner → Builder(s) → Tester → Reviewer（L3+）。每个工位 ~160×200px，包含：工匠插画（80×80，CSS 动画）+ 任务标签 + 进度条 + 💡图标（如有 TM）。当前激活的工位高亮（蓝色发光边框），已完成的显示绿色勾选。

**右侧 35%：Mirror 思考流区** — 固定在右侧，显示当前激活 Agent 的实时思考气泡。气泡从下往上堆叠（最新在底部），内容是真实 LLM 流式输出。

**底部进度轨道：** ●（完成）◉（进行中）○（待办），圆点连线，横向排列在工地最底部。

### 4.3 工匠视觉

4 张静态 PNG/SVG + CSS 动画：等待（浮动）、工作（脉冲+工具图标）、完成（弹跳+绿勾）、失败（摇晃+红叉）。工位间发光方块滑动传递（CSS transition 0.6s）。

### 4.4 Mirror System 气泡

**真实 LLM 流式输出。** Mirror 气泡以第一人称呈现工匠的真实思考过程。气泡内容是 **真实 LLM 流式输出**（通过 WebSocket 推送），不是预设文案。用户看到的"让我想想怎么拆……""正在写代码……"是 Agent 真实的 chain-of-thought 输出，字符逐个出现。

**技术实现：** 工匠开始执行任务时建立 WebSocket 连接，Agent 的 streaming response 实时推送到 Mirror 气泡区域。连接断开或超时时显示"思考中…"占位符，3秒后重连，重连失败显示"工匠正在努力工作"静态文案。

### 4.5 Teaching Moments 翻转卡片

工匠完成后，工位卡片角出现 💡 图标和边缘光晕脉冲。

**桌面端：** 鼠标悬停工位卡片 → CSS 3D 翻转（Y轴 180°，0.4s）→ 背面显示 TM 知识点 → 鼠标离开翻回正面。

每关最多 3 个 TM，不是每个工位都有。

### 4.6 TM 可发现性

无特殊首次引导。用户自然发现 hover 翻转。💡图标和边缘光晕脉冲作为视觉提示。

### 4.7 TM 内容生成规则
TM 内容由 LLM 在 Agent 完成任务时动态生成，遵循以下规则：

#### 4.7.1 写作规则
规则说明示例每句 ≤40 字短句更易理解✓ "AI 先把任务拆成小步骤。"用生活类比把代码概念比作小孩熟悉的事物✓ "就像盖房子前先画图纸"语言生动有趣避免干巴巴的定义✓ "它会一条一条找茬" ✗ "它执行验证逻辑"不出现代码术语不写 <div>、function、API 原词✓ "网页骨架" ✗ "HTML 标签"结尾强调成就"你刚才做的，和 XX 专业人士一样！"✓ "你刚才让 AI 规划任务，和项目经理一样！"
生成 Prompt 模板：
你是 CodeTown 的教育内容生成器。用户刚刚完成了一个 Agent 任务，需要生成 Teaching Moment 卡片背面内容。
Agent 类型：{agent_type}
任务描述：{task_description}
关卡：L{level}
用户年龄：10-15 岁
请生成 TM 内容，要求：
1. 标题：3-6 个字，概括这个概念
2. 正文：2-3 句话，≤80 字总计
3. 用生活中的类比解释这个概念
4. 最后一句以"你刚才……和 XX 一样！"结尾
5. 不要出现任何英文代码术语
输出格式：
标题：{title}
正文：{body}

#### 4.7.2 写作规则
各 Agent 类型的概念映射：
Agent要解释的核心概念类比方向📐 规划者任务拆解、规划画图纸、列清单、排日程🏗️ 建造者（结构）HTML / 网页骨架搭乐高、盖房子框架🏗️ 建造者（样式）CSS / 视觉美化刷油漆、贴壁纸、换衣服🏗️ 建造者（逻辑）JavaScript / 交互大脑指挥、开关控制🏗️ 建造者（技能）模块化、复用调用现成零件、请专家帮忙🔍 测试者测试、验证检查作业、找茬、质检员🔍 测试者（重试后）迭代改进做错了改、再试一次⏸️ 检查点Human-in-the-Loop关键决定由人拍板🔌 Plug外部数据调用打电话问别人、查资料📚 图书馆RAG / 带资料工作开卷考试、翻小抄
TM 判重： 如果用户在本次会话中已经看过相同概念的 TM，生成简化版："还记得吗？{概念名}！这次又用到了。"



### 4.8 揭幕时刻

构建完成→所有工匠绿勾（stagger 0.1s）→工位滑开（300ms）→iframe 淡入→confetti→「分享给朋友」+「回到小镇」两个按钮→工匠缩为底部小图标。

### 4.9 并行任务

传送带分叉两条垂直车道再合并。CSS border + position: absolute 连接器。

### 4.10 音效

**BGM：** 一首贯穿整个游戏的循环背景曲，轻快温暖风格，`<audio loop>`。

**音效（仅 2 个）：** 鼠标点击（click）+ 卡槽归位（snap）。`new Audio().play()`。

### 4.11 Token 面板

底栏：Token 用量 + ¥成本 + X/Y进度 + 渐变进度条。

### 5.1 L1 面包店新品展示页

**主题：** 为小镇面包店做一个新品展示页面。
**Template：** 选择题式。委托单预填"面包店新品展示页"，装修方案从 4 个预设风格选择（温馨手绘/简约现代/复古欧式/活力糖果），需求清单给 3 个预设选项（"展示 3 款新品""有价格标签""有联系方式"）+ 1 个自定义空位。
**工匠团队：** Planner + Builder 自动出现，用户不能修改。用户看到规划者先拆任务，建造者再执行。
**构建：** Hardcoded 3-task DAG，20-30s。单 Builder 执行。
**Teaching Moments：** TM-1 规划/Planning（AI 不是直接干活，先想好怎么做）+ TM-2 网页结构/HTML（搭骨架如盖房子）+ TM-3 样式表/CSS（换油漆方案）。

### 5.2 L2 游戏厅小游戏

**主题：** 为小镇游戏厅做一个浏览器小游戏。
**Template：** 示例式。委托单空白，给 3 个示例引导（"贪吃蛇""打砖块""猜数字"）。装修方案有预设也可自己写。需求清单全部自由输入。
**工匠团队：** 默认 Planner + Builder + Tester，用户第一次看到团队配置界面——可以调整（加减四个基础角色），但有推荐起点。
**构建：** 默认Planner 生成 DAG，5-6 工位，并行+失败重试。
**Teaching Moments：** TM-4 测试/Testing（专人找 bug）+ TM-5 迭代/Iteration（做→测→改→再测）。

### 5.3 L3 我的家（个人主页）

**主题：** 在小镇建造自己的家——一个个人主页，展示自己的兴趣、作品、或任何想表达的东西。
**Template：** 半成品模板。委托单预填"我的个人主页"，需求清单预填基础板块（"自我介绍""我的作品""联系方式"），用户可修改删减或添加。新组件（技能包、规则、流程控制、Plug）给预填模板但用户可删改。
**工匠团队：** 全部四个基础角色可用，完全自由配置。用户可以为每个角色填写 project-based prompt。
**Plug：** 2 个预制 Plug 可选（🖼️ 图片素材、🗺️ 地图）。拖入图纸即可使用，后端用平台 Key 代理。
**构建：** 审批节点暂停+批准/修改。规则自动触发可见。
**Teaching Moments：** TM-6 自动化检查/Linting（规矩自动生效）+ TM-7 Human-in-the-Loop（关键决定由人做）。

### 5.4 L4 新闻看板

**主题：** 为小镇建造一个新闻/资讯看板，能从外部获取信息并展示。
**Template：** 半成品模板。委托单预填"小镇新闻看板"，需求清单预填基础功能（"展示新闻列表""支持搜索""分类筛选"），用户可修改。
**工匠团队：** 全 9 组件自由组合。
**Plug：** L3 的 2 个预制 Plug（图片素材/地图）+ 🪄 魔法连接（用户自然语言描述需求，后端匹配白名单 API）。
**图书馆：** 粘贴文本/上传文件（.txt / .md / .doc / .docx）作为参考资料注入 Agent。单条粘贴 ≤2000 字符。
**构建：** 完整示例：委托+需求+风格+角色 project-based prompt+技能包"引用源文档"+规则"测试失败自动重试"+流程控制+Plug(魔法连接)+图书馆(参考资料)。
**Teaching Moments：** TM-8 API调用（连上互联网）+ TM-9 RAG（先查资料再回答）。毕业后→自由建造模式。

## 7. 信息架构与导航流程

### 7.1 页面结构

两层导航：

**地图（Home）→ 项目视图（设计桌 Tab + 工地 Tab）**

从地图点击建筑进入项目视图。项目视图右上角有两个 Tab（「🗒️ 设计桌」「🏗️ 工地」），默认在设计桌 Tab。工地 Tab 在"开始建造"之前为灰色不可切换。建造开始后可自由切换，但设计桌为只读。

### 7.2 进入流程

用户首次打开 → 小镇地图（四栋灰色建筑）。点击 L1 建筑 → BYOK"建筑许可证"拦截 → 输入 API Key + base URL → 进入设计桌。后续关卡如已配置 Key 则直接进入设计桌。

### 7.3 建筑状态

地图上每栋建筑有三种状态：

| 状态 | 视觉 | 交互 |
|------|------|------|
| 锁定 | 灰色 + 🔒 | 不可点击，提示"完成上一关解锁" |
| 当前可进入 | 灰色，无锁 | 可点击，进入设计桌 |
| 已完成 | 彩色 | 可点击，查看已完成项目 |

初始状态：L1 = 当前可进入，L2/L3/L4 = 锁定。完成 L1 后：L1 = 已完成，L2 = 当前可进入，L3/L4 = 锁定。依此类推。

### 7.4 场景流转规则

**正向：** 地图 → 点击建筑 → 设计桌 Tab → 填卡片 → "制定工作计划"（Meta Planner 整理卡片顺序）→ "开始建造" → 自动切换到工地 Tab → Planner 输出 DAG → 构建 → 揭幕 → 回到地图。

**Tab 切换规则：** 建造开始后，设计桌和工地 Tab 可自由切换。设计桌变为只读模式（卡片不可编辑，文件夹不可展开），顶部显示"🔒 只读模式 — 建造进行中"。用户可以查看自己的设计图纸来对照工地进度。

**揭幕后回到地图：** 建筑亮起彩色，下一关建筑变为可点击。不自动跳转下一关，让用户主动选择。

**重新构建：** 从地图再次点击已完成建筑 → 可选择"查看项目"或"重新建造" → 重新建造进入空白设计桌。

### 7.5 中断恢复策略

| 场景 | 行为 |
|------|------|
| 设计桌填写中退出 | **保存设计桌内容**，再次进入恢复已填卡片 |
| 工地构建中退出 | **设计桌内容保存**，但构建中断不支持恢复。再次进入时设计桌 Tab 高亮，工地 Tab 灰色，需重新点击"开始建造" |
| 构建已完成 | 项目持久化，地图上建筑保持彩色 |

> **设计决策：** v6.5 支持设计桌内容持久化。工地构建进度不保存——用户需要重新开始建造，但不需要重新填写卡片。

### 7.6 BYOK 时机

首次点击建筑时触发 BYOK 拦截（而非进入设计桌后或按"制定工作计划"时），避免用户花时间填写图纸后才发现无法构建。API Key + base url存于 React state（+ sessionStorage），跨关卡复用，关标签清除。"建筑许可证"引导（家长注册→创建密钥→粘贴）。Key 格式校验：`sk-` 前缀（Opencode API Key）。面包店约¥0.2。

### 9.5 构建流水线

**Pre-Plan（Meta Planner，用户可见结果）：** 用户点击"制定工作计划"后运行。Meta Planner 整理图纸卡片——按卡槽从上到下的顺序排列，输出结构化的项目输入。完成后按钮切换为"开始建造"。L1 hardcoded 顺序（委托单→装修→需求），L2+ Sonnet 动态整理。

**Plan（Planner Agent）：** 工地第一个工位。Planner 工匠拿到结构化输入后，拆解为具体实现步骤，输出 Task DAG。L1 hardcoded DAG（固定 3 步）。L2+ Sonnet Planner Agent 动态生成。流程控制指令通过 `<flow_control_constraints>` 注入，Planner 在生成 DAG 时考虑这些约束。

**Execute：** 拓扑顺序。system prompt = role + project-based prompt + skills(matching target) + always active rules。流式 WebSocket。Mirror 气泡。on_task_done → 每个子任务完成都触发对应规则 → Reviewer（如启用）。"让我看看" → 暂停等用户。

**Test：** Tester 工匠代码审查验证。FAIL → on_test_fail 规则触发 → Builder 重试(max_retries)。

**Deploy：** before_deploy 规则 → 存HTML → share_slug → town_state → build_complete → 揭幕。

### 9.6 Mirror System

Mirror 气泡显示 Agent 的真实 chain-of-thought 输出，通过 WebSocket 流式推送。

```typescript
// WebSocket 消息类型
interface MirrorChunk {
  type: 'mirror_chunk';
  agentId: string;
  content: string;  // 增量文本
  isComplete: boolean;
}

// 前端渲染逻辑
const MirrorBubble = ({ agentId }: { agentId: string }) => {
  const [text, setText] = useState('');
  
  useEffect(() => {
    ws.on('mirror_chunk', (chunk: MirrorChunk) => {
      if (chunk.agentId === agentId) {
        setText(prev => prev + chunk.content);
      }
    });
  }, [agentId]);
  
  return <div className="mirror-bubble">{text || '思考中…'}</div>;
};

// 后端从 Agent streaming response 提取 thinking 内容
const streamToMirror = (agentStream: ReadableStream, ws: WebSocket, agentId: string) => {
  const reader = agentStream.getReader();
  const processChunk = async () => {
    const { done, value } = await reader.read();
    if (done) {
      ws.send(JSON.stringify({ type: 'mirror_chunk', agentId, content: '', isComplete: true }));
      return;
    }
    // 提取 thinking/reasoning 部分发送到 Mirror
    const thinkingContent = extractThinking(value);
    if (thinkingContent) {
      ws.send(JSON.stringify({ type: 'mirror_chunk', agentId, content: thinkingContent, isComplete: false }));
    }
    processChunk();
  };
  processChunk();
};
```

**降级处理：** WebSocket 断开时显示"思考中…"占位符，3秒后重连，重连失败显示静态文案"工匠正在努力工作"。

### 9.10 异常路径

#### API 相关（统一处理）

| 场景 | UX 表现 | 技术处理 |
|------|---------|---------|
| API Key 为空 | 点击建筑时模态框拦截，不可跳过 | 阻断进入设计桌 |
| API Key 格式不符（不以 sk- 开头） | 输入框下方红字提示 | 阻断，重新输入 |
| API Key 无效（401）| 弹模态框"许可证无效，请重新输入" | 暂停不终止，重新输入后继续 |
| API Key 额度用完（429） | 弹模态框"许可证无效，请重新输入" | 暂停不终止，重新输入后继续 |
| API 服务报错 | 弹模态框"许可证无效，请重新输入" | 暂停不终止，重新输入后继续 |
| 单次 Agent 调用超时（>60s） | 工匠摇晃 + 给用户"重试/跳过" | 自动重试 1 次（2s），仍失败交由用户决定 |
|仅有API key但是没有base URL|弹模态框"请输入base url"|

> **统一逻辑：** 401 / Key 无效 / Key 不存在 / API 服务报错 → 全部判定为"许可证无效" → 弹模态框重新输入 → 直到成功。

#### 构建流程

| 场景 | UX 表现 | 技术处理 |
|------|---------|---------|
| 必填卡槽未填 | "制定工作计划"按钮置灰 + 未填卡槽高亮闪动 | 前端校验，不发请求 |
| Agent 返回空内容或格式错误 | 工匠摇晃 + "重试中" | 自动重试 1 次（2s），仍失败给用户"重试/跳过" |

| MetaPlanner 处理失败 | L1：用户无感（hardcoded fallback）；L2-L4：工匠摇晃 + 给用户"重试/停止" | L1 降级为 hardcoded DAG；L2-L4 自动重试 1 次，仍失败交由用户决定 |

#### 网络

| 场景 | UX 表现 | 技术处理 |
|------|---------|---------|
| WebSocket 断开 | 黄色横幅"重连中…"，保留已完成工位状态 | 指数退避重试 5 次 + 状态快照 |

#### L3 特有异常

| 场景 | UX 表现 | 技术处理 |
|------|---------|---------|
| 并行分支中单条失败 | 该分支工位标记失败，其他分支继续 | 合并节点汇总时失败分支视为跳过 |

| 预制 Plug 调用失败 | 工匠气泡"外部服务暂时不可用" + "重试/跳过" | 用户决定 |
| 规则触发死循环 | 全局 max 3 次循环后强制停止 + 提示用户 | 循环计数器，超限中断 |

### 14.26 异常与恢复（Error Recovery）

| AC | Given | When | Then |
|----|-------|------|------|
| ERR-1 API 错误统一处理 | Key 格式通过但 API 返回任何错误（401/429/服务报错/不存在） | 工地启动或构建中 API 调用失败 | 工地暂停，弹模态框"许可证无效，请重新输入"，输入后继续，不终止 |
| ERR-2 Agent 超时 | 单次 Agent 调用 >60s | 构建中 | 自动重试 1 次（2s 延迟），仍失败 → 工匠摇晃 + "重试"/"跳过"按钮 |
| ERR-3 Agent 空内容 | Agent 返回空内容或格式错误 | 构建中 | 自动重试 1 次（2s），仍失败 → "重试"/"跳过" |
| ERR-4 Tester 重试上限 | Tester 失败重试 3 次仍不通过 | 第 3 次失败后 | 暂停 → "再试一次"/"跳过这个任务"/"停止构建"三按钮，不自动跳过 |
| ERR-5 输出过大 | 输出 HTML >200KB | Deploy 阶段 | toast"项目太大了，已简化" + 截断至 200KB |
| ERR-6 MetaPlanner 失败-L1 | L1 MetaPlanner 处理失败 | 点击"制定工作计划" | 用户无感，降级为 hardcoded fallback |
| ERR-7 MetaPlanner 失败-L2+ | L2+ MetaPlanner 处理失败 | 点击"制定工作计划" | 自动重试 1 次，仍失败 → 工匠摇晃 + "重试"/"停止" |
| ERR-8 WebSocket 断开 | 构建过程中网络中断 | WebSocket 断开 | 黄色横幅"重连中…"，保留已完成工位状态，指数退避重试 5 次 |
| ERR-9 WebSocket 重连失败 | 5 次重试全部失败 | 最后一次失败后 | 提示"网络不稳定，请刷新重试"，构建进度丢失 |
| ERR-10 并行分支单条失败 | L3+ 并行分支中一条失败 | 构建中 | 该分支标记失败，其他分支继续执行；合并时失败分支视为跳过 |
| ERR-11 审批节点超时 | L3+ 审批节点等待用户 | 持续等待 | 不自动超时，工位显示"等待你的决定…"，等用户操作 |
| ERR-12 预制 Plug 失败 | L3+ 图片素材或地图 Plug 调用失败 | 构建中 | 工匠气泡"外部服务暂时不可用" + "重试"/"跳过" |
| ERR-13 魔法连接无法匹配 | L4 魔法连接描述无法匹配任何 API 类别 | 构建中 | Agent 收到"暂时不支持这种连接"提示，尝试其他方案或告知用户 |
| ERR-14 规则死循环 | L3+ 规则触发形成循环 | 循环执行第 3 次 | 强制停止 + 提示"规则循环了，已自动停止" |
| ERR-15 中断恢复-设计桌 | 用户在设计桌填写中退出 | 再次进入该关卡 | 自动加载已保存草稿，恢复已填卡片 |
| ERR-16 中断恢复-工地 | 用户在工地构建中退出 | 再次进入该关卡 | 设计桌 Tab 高亮，已填卡片恢复，工地 Tab 灰色（构建进度丢失），需重新点击"开始建造" |
| ERR-17 中断恢复-已完成 | 用户完成构建后退出 | 再次打开游戏 | 项目持久化，地图上建筑保持彩色 |
